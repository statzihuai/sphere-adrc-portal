# SPHERE ADRC Portal — Engineer Brief
**Version:** June 2026 · Prepared for incoming engineer

## What you're building on

The Stanford ADRC Synthetic Research Data Hub is a public-facing research portal that lets scientists explore a privacy-preserving synthetic twin of the Stanford Alzheimer's Disease Research Center dataset (644 participants, 9 data modalities, 206,000+ features), generated by the SPHERE algorithm.

**The portal uses SPHERE×2 data** — the result of applying two rounds of SPHERE rotation to the joint data matrix. The pipeline generates k=1 through k=5 for research comparison, but `adrc_data.js` (the compiled data file the portal loads) is built specifically from the k=2 output (`sphere_xk/x2/`). This is set in `scripts/run_all.py` step 5.

**`adrc_data.js` is a curated subset, not the full SPHERE×2 data.** The `portal/data/sphere/*.csv` files contain the complete SPHERE×2 output, but `portal/scripts/inject_data.py` selects only a browser-feasible subset when compiling `adrc_data.js`:

| Modality | In `adrc_data.js` | Full SPHERE×2 CSV |
|---|---|---|
| demographics | 5 core columns | 30 columns |
| cognitive | ✅ complete (~74 cols) | same |
| biomarkers | ✅ complete (3 cols) | same |
| imaging | ✅ complete (~577 cols) | same |
| wgs | ✅ complete (~167 SNPs) | same |
| scrna | ⚠️ 21 marker genes only | ~193,326 gene-cell columns |
| proteomics_csf | ⚠️ ~5 curated aptamers | ~5,284 aptamers |
| proteomics_plasma | ⚠️ ~5–18 curated aptamers | ~6,900 aptamers |

The full CSVs in `portal/data/sphere/` are intended for direct download (as a zip), not browser loading. scRNAseq and proteomics are too large for browser memory at full width.

The portal is live at: `https://web.stanford.edu/~zihuai/adrc-sphere/`

The codebase is a **single `index.html`** (~4,500 lines, no build system, no framework). This was intentional for static hosting on Stanford AFS. The data is compiled into three JS files (`adrc_data.js` ~11MB, `adrc_dd.js` ~3MB, `adrc_validation.js` ~230KB) that load with the page.

The portal includes a **SPHERE AI Agent** tab — an embedded AI analyst powered by Claude. Currently it uses BYOK (users paste their own Anthropic API key). Your job is to replace this with a proper backend.

## Phase 1 — Backend foundation (start here)

### 1a. Anthropic streaming proxy
This is the most technically critical piece. Get it right first.

The current portal uses SSE streaming with `stream: true`. The backend must:
- Accept a POST with `{ messages, model, system }` from the browser
- Forward to Anthropic's `/v1/messages` with SPHERE's API key
- **Stream the SSE response chunks back to the browser in real time** — do not buffer
- Log all four token fields from the final `message_delta` event:
  - `input_tokens`
  - `cache_read_input_tokens`
  - `cache_creation_input_tokens`
  - `output_tokens`
- Deduct credits based on `(input_tokens + cache_read_input_tokens) × $3/1M + output_tokens × $15/1M` (Sonnet rates — bill at retail regardless of what SPHERE actually pays)

```
Anthropic response fields to log per request:
  usage.input_tokens                — fresh input, charged at full rate
  usage.cache_read_input_tokens     — cached, billed to user at full rate, SPHERE pays 10x cheaper
  usage.cache_creation_input_tokens — first write to cache
  usage.output_tokens               — output
```

### 1b. Auth
- Email/password signup + login
- OAuth (Google) — nice to have for researchers
- JWT with refresh tokens
- On signup: create account, add $10 trial credits

### 1c. Credit ledger
- Table: `(user_id, timestamp, delta, balance_after, description, request_id)`
- Atomic deduction — check balance before forwarding to Anthropic
- Return 402 if insufficient credits (browser should show "Add credits")

### 1d. Stripe integration
- Card on file (Stripe Customer + PaymentMethod)
- $29/month subscription via Stripe Subscriptions
- Credit pack purchases (one-time payments)
- Webhooks for subscription events

The current AI agent loads data from `window.ADRC_DATA` — a pre-compiled browser-memory subset (~11MB). The end product must give the AI access to all 206,000+ features.

**Key design decision: keep Python execution in the browser (Pyodide).** All analysis runs locally on the user's machine. Stanford's server only serves CSV/parquet files. Privacy is preserved — Stanford sees which datasets were requested, but never the analysis or results.

**How it works:**

const resp = await fetch('https://api.sphere.io/data/sphere2/' + name + '.csv',

**What does NOT change:** `run_python`, all tool definitions, Pyodide setup, figures, streaming — everything else is identical.

## Phase 4 — Shared prompt caching + token billing

This is the margin engine. Read section 3 of `sphere_business_model.html` before implementing this.

### Why shared caching matters

With a centralized SPHERE API key, **all users share the same Anthropic prompt cache**. When user A sends a message, the system prompt gets cached under SPHERE's key. When user B sends a message 30 seconds later, their request hits that same warm cache — Anthropic charges SPHERE the discounted cache-read rate, but SPHERE bills user B at the full retail rate. The spread is pure margin.

With BYOK (each user's own key), there is no shared cache and no margin. This only works with a centralized key.

```
User A  ──┐
User B  ──┤──→  SPHERE Backend (one API key)  ──→  Anthropic
User C  ──┘         ↑ shared cache namespace
                    all hit same warm cache
```

### What gets cached

The system prompt is already marked `cache_control: {type: "ephemeral"}` in the current frontend. The backend must preserve this when forwarding to Anthropic. Do not strip it.

**Currently cached (~900 tokens):**
- SPHERE system prompt (dataset descriptions, instructions, data scope notes)
- Tool definitions (`load_dataset`, `run_python`, etc.)

**Target cached context (~4,500 tokens — expand before launch):**
- Everything above, plus a curated Stanford clinical/statistical knowledge base (~3,600 tokens)
- This increases cache value: more tokens cached = higher margin per query

**Never cache:** The 151,000-token full data dictionary. It lives in Pyodide memory only and never touches the API.

### Token accounting — exact schema

Anthropic returns four token counts in every response. Log all four, every request, no exceptions:

```
usage.input_tokens                — fresh uncached input tokens
usage.cache_creation_input_tokens — tokens written to cache this request (1.25× cost to SPHERE)
usage.cache_read_input_tokens     — tokens read from cache (0.1× cost to SPHERE)
usage.output_tokens               — output tokens
```

**Database table: `api_usage_log`**
```sql
CREATE TABLE api_usage_log (
  id                          SERIAL PRIMARY KEY,
  user_id                     INTEGER REFERENCES users(id),
  session_id                  TEXT,
  timestamp                   TIMESTAMPTZ DEFAULT now(),
  model                       TEXT,
  input_tokens                INTEGER,   -- fresh input
  cache_creation_tokens       INTEGER,   -- written to cache
  cache_read_tokens           INTEGER,   -- read from cache
  output_tokens               INTEGER,
  billed_input_tokens         INTEGER,   -- = input + cache_creation + cache_read
  billed_output_tokens        INTEGER,   -- = output_tokens
  user_charge_usd             NUMERIC(10,6),
  sphere_cost_usd             NUMERIC(10,6),
  margin_usd                  NUMERIC(10,6)
);
```

### Billing formula

Bill the user as if no caching exists — full retail on every token sent:

```
billed_input_tokens  = input_tokens + cache_creation_tokens + cache_read_tokens
billed_output_tokens = output_tokens

user_charge = billed_input_tokens × INPUT_RATE + billed_output_tokens × OUTPUT_RATE
```

SPHERE's actual cost to Anthropic:
```
sphere_cost = input_tokens            × INPUT_RATE
            + cache_creation_tokens   × INPUT_RATE × 1.25   ← 25% surcharge on cache writes
            + cache_read_tokens       × INPUT_RATE × 0.10   ← 90% discount on cache reads
            + output_tokens           × OUTPUT_RATE
```

Margin per request:
```
margin = user_charge - sphere_cost
```

**Concrete example (Sonnet, typical query after cache is warm):**

| Token type | Count | User pays | SPHERE pays |
|---|---|---|---|
| `input_tokens` (fresh) | 500 | $0.00150 | $0.00150 |
| `cache_creation_tokens` | 0 | $0 | $0 |
| `cache_read_tokens` | 4,500 | $0.01350 | $0.00135 |
| `output_tokens` | 1,000 | $0.01500 | $0.01500 |
| **Total** | | **$0.03000** | **$0.01785** |
| **Margin** | | | **~40%** |

On first request (cold cache, writing 4,500 tokens):

| Token type | Count | User pays | SPHERE pays |
|---|---|---|---|
| `input_tokens` | 5,000 | $0.01500 | $0.01500 |
| `cache_creation_tokens` | 4,500 | $0.01350 | $0.01688 |
| `output_tokens` | 1,000 | $0.01500 | $0.01500 |
| **Total** | | **$0.04350** | **$0.04688** |
| **Margin** | | | **−8% (loss)** |

Cache writes are slightly loss-making — this is expected and fine. SPHERE recovers it on the next request from any user. At steady traffic the economics are strongly positive.

### Rates to use (Sonnet — update if model pricing changes)

```python
INPUT_RATE  = 3.00 / 1_000_000   # per token
OUTPUT_RATE = 15.00 / 1_000_000  # per token
# SPHERE's negotiated rates (15% volume discount):
SPHERE_INPUT_RATE  = 2.55 / 1_000_000
SPHERE_OUTPUT_RATE = 12.75 / 1_000_000
CACHE_WRITE_MULT   = 1.25   # applied to SPHERE_INPUT_RATE
CACHE_READ_MULT    = 0.10   # applied to SPHERE_INPUT_RATE
```

Store model rates in a config table, not hardcoded — Anthropic changes pricing.

### Cache warming heartbeat

Anthropic's cache TTL is **5 minutes**. If no request arrives for 5 minutes, the cache expires and the next user pays the cold-cache write cost. At low traffic volumes, implement a heartbeat:

```python
# Run every 4 minutes (cron or background task)
async def warm_cache():
    await anthropic.messages.create(
        model=DEFAULT_MODEL,
        max_tokens=1,
        system=[{"type": "text", "text": SPHERE_SYSTEM_PROMPT,
                 "cache_control": {"type": "ephemeral"}}],
        messages=[{"role": "user", "content": "ping"}]
    )
    # Cost: ~$0.001 per ping. Saves cache miss cost on next real user request.
```

Stop the heartbeat when no users have been active for >30 minutes.

## Phase 5 — Stripe payment integration

### Overview of money flows

```
User signs up       → $10 trial credits added (no card)
Trial exhausted     → prompt to add card
Pay-as-you-go       → purchase credit packs ($10, $25, $50)
Monthly subscriber  → $29/month via Stripe Subscription
                      includes: 10M cell generation allowance
                                10M cell eval & certify allowance
                                $20 AI credits
AI usage            → deducted from credit balance in real time
                      (billed at Claude retail rate — see Phase 4)
```

### Stripe objects to create

| Object | Purpose |
|---|---|
| `Customer` | One per SPHERE account. Created at signup. |
| `PaymentMethod` | Card on file. Attached to Customer. |
| `Subscription` | $29/month plan. Optional — user stays pay-as-you-go if they don't subscribe. |
| `PaymentIntent` | One-time credit pack purchase. |
| `Product` + `Price` | Define the $29/month plan and credit pack SKUs in Stripe dashboard. |
| `WebhookEndpoint` | Receives Stripe events (payment success, subscription changes, failures). |

### Database schema additions

```sql
-- Billing profile per user
CREATE TABLE billing (
  user_id              INTEGER PRIMARY KEY REFERENCES users(id),
  stripe_customer_id   TEXT UNIQUE,          -- cus_xxx
  stripe_sub_id        TEXT,                 -- sub_xxx (null if pay-as-you-go)
  sub_status           TEXT,                 -- active | past_due | canceled | null
  sub_period_end       TIMESTAMPTZ,          -- next renewal date
  credit_balance_usd   NUMERIC(10,4) DEFAULT 0,  -- current spendable balance
  trial_used           BOOLEAN DEFAULT FALSE
);

-- Every credit movement (purchases, usage, refunds, trial grants)
CREATE TABLE credit_ledger (
  id              SERIAL PRIMARY KEY,
  user_id         INTEGER REFERENCES users(id),
  timestamp       TIMESTAMPTZ DEFAULT now(),
  delta_usd       NUMERIC(10,6),             -- positive = credit added, negative = deducted
  balance_after   NUMERIC(10,4),
  type            TEXT,                      -- trial_grant | credit_pack | subscription_grant
                                             -- ai_usage | generation | eval_certify | refund
  description     TEXT,
  stripe_pi_id    TEXT,                      -- PaymentIntent id if applicable
  api_usage_id    INTEGER REFERENCES api_usage_log(id)  -- link to AI usage record
);
```

### User flows

**1. Signup → trial credits**
```
POST /auth/signup
  → create users row
  → create Stripe Customer (no card yet)
  → insert billing row
  → if not trial_used:
      INSERT credit_ledger (delta=+10.00, type='trial_grant')
      UPDATE billing SET trial_used=true, credit_balance_usd=10.00
  → return JWT
```

**2. Add card (after trial exhausted)**
```
Frontend: Stripe.js → stripe.createPaymentMethod() → sends paymentMethod.id to backend
POST /billing/add-card { payment_method_id }
  → stripe.paymentMethods.attach(pm_id, { customer: stripe_customer_id })
  → stripe.customers.update(cus_id, { invoice_settings: { default_payment_method: pm_id } })
  → return { success: true }
```

**3. Purchase credit pack (one-time)**
```
POST /billing/purchase-credits { amount_usd: 25 }
  → stripe.paymentIntents.create({
        amount: 2500,  // cents
        currency: 'usd',
        customer: stripe_customer_id,
        payment_method: default_pm,
        confirm: true,
        off_session: true
    })
  → on success:
      INSERT credit_ledger (delta=+25.00, type='credit_pack', stripe_pi_id=pi_xxx)
      UPDATE billing SET credit_balance_usd += 25.00
```

**4. Subscribe to $29/month plan**
```
POST /billing/subscribe
  → stripe.subscriptions.create({
        customer: stripe_customer_id,
        items: [{ price: PRICE_ID_29_MONTHLY }],
        payment_settings: { payment_method_types: ['card'] }
    })
  → on webhook invoice.payment_succeeded:
      INSERT credit_ledger (delta=+20.00, type='subscription_grant', description='$20 AI credits')
      INSERT credit_ledger (delta=+0, type='subscription_grant', description='10M cell generation allowance')
      UPDATE billing SET sub_status='active', sub_period_end=...
      UPDATE billing SET credit_balance_usd += 20.00
      (cell allowances tracked separately in a generation_allowance table)
```

**5. AI usage deduction (every API proxy call)**
```python
# In the Anthropic proxy, after streaming completes and usage is known:
charge = (billed_input_tokens * INPUT_RATE) + (billed_output_tokens * OUTPUT_RATE)

async with db.transaction():
    balance = await db.fetchval("SELECT credit_balance_usd FROM billing WHERE user_id=$1 FOR UPDATE", user_id)
    if balance < charge:
        raise InsufficientCreditsError()
    new_balance = balance - charge
    await db.execute("UPDATE billing SET credit_balance_usd=$1 WHERE user_id=$2", new_balance, user_id)
    await db.execute("""
        INSERT INTO credit_ledger (user_id, delta_usd, balance_after, type, api_usage_id)
        VALUES ($1, $2, $3, 'ai_usage', $4)
    """, user_id, -charge, new_balance, usage_log_id)
```

**Important:** Check balance BEFORE forwarding to Anthropic (return 402 if insufficient), then deduct AFTER usage is confirmed. Use a database transaction with `FOR UPDATE` to prevent race conditions from concurrent requests.

### Stripe webhooks to handle

| Event | Action |
|---|---|
| `invoice.payment_succeeded` | Grant monthly subscription credits, update `sub_period_end` |
| `invoice.payment_failed` | Set `sub_status='past_due'`, email user, restrict AI access after grace period |
| `customer.subscription.deleted` | Set `sub_status='canceled'`, remove cell allowances |
| `payment_intent.succeeded` | Confirm credit pack purchase if not already handled synchronously |
| `payment_intent.payment_failed` | Log failure, notify user |

Verify webhook signatures using `stripe.webhooks.constructEvent(payload, sig, WEBHOOK_SECRET)`. Reject any webhook that fails signature check.

### Security notes

- Never log full card numbers or CVVs — Stripe handles this, your backend never sees raw card data
- Store only `stripe_customer_id` and `stripe_subscription_id` in your database
- Use Stripe's test mode (`sk_test_...`) during development, switch to live key (`sk_live_...`) at launch
- Set up separate Stripe webhook endpoints for test and production environments
- Idempotency: use `idempotencyKey` on PaymentIntent creation to prevent double-charges on retries

### Frontend additions needed in `index.html`

- Load Stripe.js: `<script src="https://js.stripe.com/v3/"></script>`
- Replace "Connect API key" flow with sign-in + credit balance display
- Add "Add card" modal using Stripe Elements (PCI-compliant card input)
- Show credit balance in the agent status bar, update after each AI request
- Show low-balance warning when balance drops below $2.00

## Key constraints

1. **Pyodide stays client-side.** Python execution, DataFrame operations, matplotlib figures — all run in the user's browser. The backend never receives data, only conversation messages.

2. **Streaming is non-negotiable.** The AI typing effect depends on SSE streaming. Any backend that buffers and returns a full response will break the UX. Test SSE proxying explicitly.

3. **Token accounting must be exact.** The billing model depends on logging all four token fields from every Anthropic response. Missing `cache_read_input_tokens` means undercharging users.

4. **The portal is one file.** `index.html` is ~4,500 lines. It will eventually need to be split into components, but for now work with it as-is to minimize refactoring risk during the backend transition.

5. **No real patient data.** The `real/` directory is not included and should never be. The `sphere/` CSVs are synthetic. Treat all data in this repo as public.
